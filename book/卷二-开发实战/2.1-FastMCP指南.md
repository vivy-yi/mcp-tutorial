## 一、背景故事：一个真实的开发困境

### 1.1 传统 MCP 开发的痛点

想象这样的场景：你正在为一家科技公司构建内部的 AI 助手，需要让它能够访问公司的代码仓库、文档系统和工单系统。

传统的做法是这样的：

```
┌─────────────────────────────────────────────────────────────────┐
│  传统 MCP 服务器开发流程                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第1步：定义工具接口                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ def get_repos(): ...                                   │   │
│  │ def get_file_content(path): ...                        │   │
│  │ def create_issue(title, body): ...                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  第2步：实现 JSON-RPC 处理层                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 编写 request/response 序列化                            │   │
│  │ 处理方法路由                                            │   │
│  │ 实现错误处理                                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  第3步：实现传输层                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 选择 Stdio 还是 HTTP？                                   │   │
│  │ 处理连接复用                                            │   │
│  │ 实现心跳检测                                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  第4步：配置客户端集成                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Claude Desktop 配置                                      │   │
│  │ Cursor 配置                                              │   │
│  │ 自定义客户端适配                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  结果：80% 的时间花在基础设施代码上，                           │
│       真正业务逻辑（工具实现）只占 20%！                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

这就是 FastMCP 诞生的背景：**让开发者专注于业务逻辑，把基础设施交给框架**。

### 1.2 FastMCP 的设计哲学

FastMCP 遵循一个核心原则：

> **"约定优于配置"** + **"最小惊讶原则"**

这意味着：

```
┌─────────────────────────────────────────────────────────────────┐
│                    FastMCP 设计原则                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 极简 API                                                  │
│     用最少的代码表达最多的功能                                  │
│                                                                 │
│     ❌ 传统: 50 行配置 + 100 行实现                           │
│     ✅ FastMCP: 10 行代码                                      │
│                                                                 │
│  2. 渐进式复杂度                                              │
│     简单场景简单写，复杂场景有进阶 API                          │
│                                                                 │
│     @app.tool()                  # 基础用法                     │
│     @app.tool(schema=...)       # 中级用法                     │
│     @app.tool(..., middleware)   # 高级用法                    │
│                                                                 │
│  3. 类型安全优先                                              │
│     利用 Python 类型提示减少运行时错误                           │
│                                                                 │
│  4. 错误即文档                                                │
│     清晰的错误信息是最好的使用文档                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、核心概念

### 3.1 工具 (Tools) 的本质

在深入代码之前，我们需要理解一个关键问题：**工具在 MCP 协议中到底是什么？**

```
┌─────────────────────────────────────────────────────────────────┐
│                       工具的双重身份                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  从协议角度看                                                   │
│  ─────────────                                                  │
│                                                                 │
│  工具 = 描述 (Schema) + 处理逻辑 (Handler)                     │
│                                                                 │
│  ┌─────────────────┐         ┌─────────────────┐              │
│  │    Schema      │         │    Handler      │              │
│  ├─────────────────┤         ├─────────────────┤              │
│  │ name: str      │         │ def call():     │              │
│  │ description:    │         │     ...         │              │
│  │ inputSchema:    │         │     return      │              │
│  │   type: object │         │                  │              │
│  │   properties:   │         │                  │              │
│  └─────────────────┘         └─────────────────┘              │
│                                                                 │
│  协议定义了"是什么"（描述），                                      │
│  FastMCP 实现了"怎么做"（处理）                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 工具描述的生成原理

FastMCP 最强大的特性之一是能够从函数签名自动生成完整的 JSON Schema：

```
┌─────────────────────────────────────────────────────────────────┐
│              类型注解到 JSON Schema 的转换                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Python 类型                 JSON Schema                          │
│  ──────────                ────────────                         │
│                                                                 │
│  str                   →  {"type": "string"}                  │
│                                                                 │
│  int                   →  {"type": "integer"}                 │
│                                                                 │
│  float                 →  {"type": "number"}                  │
│                                                                 │
│  bool                  →  {"type": "boolean"}                 │
│                                                                 │
│  List[str]             →  {"type": "array",                  │
│                             "items": {"type": "string"}}      │
│                                                                 │
│  Dict[str, int]        →  {"type": "object",                 │
│                             "additionalProperties":             │
│                               {"type": "integer"}}            │
│                                                                 │
│  Optional[str]         →  {"type": "string"} (可空)          │
│                                                                 │
│  Literal["a", "b"]     →  {"enum": ["a", "b"]}               │
│                                                                 │
│  Enum.MyEnum           →  {"enum": ["A", "B", "C"]}          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

配合 Pydantic 模型时，转换更加丰富：

┌─────────────────────────────────────────────────────────────────┐
│              Pydantic 模型到 JSON Schema 的转换                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  class User(BaseModel):                                         │
│      name: str              →  {"type": "string",              │
│      age: int = 18              "default": 18}                │
│      email: EmailStr         →  {"type": "string",              │
│                                   "format": "email"}           │
│      tags: List[str]        →  {"type": "array",              │
│                                   "items": {"type": "string"}}│
│                                                                         │
│  字段描述也自动转换                                              │
│  ────────────────                                               │
│                                                                         │
│  name: str = Field(description="用户姓名")                         │
│                        ↓                                         │
│  {"description": "用户姓名"}                                    │
│                                                                         │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 请求处理流程

当客户端发送请求时，FastMCP 内部发生了什么：

```
┌─────────────────────────────────────────────────────────────────┐
│                    请求处理详细流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────┐                                              │
│  │  收到消息     │                                              │
│  └──────┬───────┘                                              │

│  │                                                          │   │
│  │  验证: jsonrpc="2.0", method 存在, params 格式正确      │   │
│  └──────────────────────────┬───────────────────────────────┘   │
│                             │                                    │
│                             ▼                                    │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  Step 2: 方法路由                                       │   │
│  │                                                          │   │
```

## 六、核心实现解析

> **注意**：以下代码仅用于讲解原理，完整的可运行代码请参考附录。

### 6.1 装饰器的实现原理

FastMCP 的核心魔法在于装饰器。理解它的实现，能帮助你更好地使用框架：

```
┌─────────────────────────────────────────────────────────────────┐
│                    装饰器工作原理                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  @app.tool() 的内部逻辑：                                       │
│                                                                 │
│  def tool(self, name=None, description=None):                   │
│      """工具装饰器"""                                           │
│                                                                 │
│      def decorator(func):                                       │
│          # 1. 从函数签名提取信息                                 │
│          tool_name = name or func.__name__                     │
│          tool_description = description or func.__doc__        │
│          input_schema = self._generate_schema(func)             │
│                                                                 │
│          # 2. 创建工具定义                                       │
│          tool_def = ToolDefinition(                             │
│              name=tool_name,                                    │
│              description=tool_description,                       │
│              input_schema=input_schema                          │
│          )                                                      │
│                                                                 │
│          # 3. 注册到工具注册表                                    │
│          self._tool_registry.register(tool_def, func)          │
│                                                                 │
│          # 4. 返回原函数（不修改行为）                           │
│          return func                                            │
│                                                                 │
│      return decorator                                           │
│                                                                 │
│  这个设计的关键在于：                                            │
│  • 装饰器不修改函数的运行时行为                                  │
│  • 只在注册阶段提取元信息                                        │
│  • 运行时完全透明                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
│     ─────────────────                                          │

│                                                                 │
│  ✅ 文档字符串完整                                              │
```

### 7.2 错误处理模式

```
┌─────────────────────────────────────────────────────────────────┐
│                    推荐错误处理模式                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  推荐：返回结构化错误                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @app.tool()                                          │   │
│  │  def divide(a, b):                                   │   │
│  │      if b == 0:                                       │   │
│  │          return {                                     │   │
│  │              "success": False,                       │   │
│  │              "error": "除数不能为零"                  │   │
│  │          }                                           │   │
│  │      return {"success": True, "result": a / b}      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  不推荐：抛出异常                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @app.tool()                                          │   │
│  │  def divide(a, b):                                   │   │
│  │      if b == 0:                                       │   │
│  │          raise ValueError("除数不能为零")              │   │
│  │      return a / b                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  原因：抛出异常会被 MCP 协议包装成通用错误，                     │
│        返回结构化字典可以提供更精确的错误信息                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
