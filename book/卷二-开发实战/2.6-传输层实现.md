## 一、引子：数据的"最后一公里"

### 1.1 一个有趣的类比

想象你在使用 Claude Desktop 帮你操作 GitHub。当 Claude 说"让我查看一下你的仓库"时，这个请求是如何到达 GitHub MCP 服务器的？

```
┌─────────────────────────────────────────────────────────────────┐
│                      MCP 请求的旅程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Claude Desktop                                                 │
│       │                                                         │
│       │  "查看仓库" ← 用户的自然语言                             │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  思考过程：需要调用 GitHub 工具                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       │ JSON-RPC 格式                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  {                                                        │   │
│  │    "method": "tools/call",                             │   │
│  │    "params": {"name": "list_repos", ...}             │   │
│  │  }                                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       │ ⚠️ 问题来了：这封信怎么送达？                         │
│       ▼                                                         │
│  ════════════════════════════════════════════════════════════   │
│                    传输层——数据的"最后一公里"                    │
│  ════════════════════════════════════════════════════════════   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

传输层要解决的问题就是：**"如何把 JSON-RPC 消息从 A 点送到 B 点"**。

### 1.2 传输层的本质

```
┌─────────────────────────────────────────────────────────────────┐
│                    传输层在协议栈中的位置                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  应用层     ┌─────────────────────────────────────────────┐     │
│             │  MCP 协议 (JSON-RPC 消息)                  │     │
│             │  • tools/call                               │     │
│             │  • resources/read                           │     │
│             │  • prompts/get                              │     │
│             └────────────────────┬────────────────────────┘     │
│                                  │                              │
│  ────────────────────────────────┼───────────────────────────   │
│                                  ▼                              │
│  传输层     ┌─────────────────────────────────────────────┐     │
│             │  数据传输方式                                │     │
│             │  • Stdio (标准输入输出)                      │     │
│             │  • HTTP + SSE                              │     │
│             │  • Streamable HTTP                          │     │
│             └────────────────────┬────────────────────────┘     │
│                                  │                              │
│  ────────────────────────────────┼───────────────────────────   │
│                                  ▼                              │
│  网络层     ┌─────────────────────────────────────────────┐     │
│             │  TCP/IP, WebSocket, SSL/TLS                 │     │
│             └─────────────────────────────────────────────┘     │
│                                                                 │
│  关键理解：                                                    │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  JSON-RPC 定义了"说什么"                               │ │
│  │  传输层定义了"怎么说"                                   │ │
│  │                                                           │ │
│  │  两者正交——可以自由组合                                  │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 二、核心概念：三种传输方式

### 2.1 Stdio 传输：程序的" stdin/stdout"

**场景**：当你本地开发 MCP 服务器时，最常用的方式。

```
┌─────────────────────────────────────────────────────────────────┐
│                      Stdio 传输原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  原理：使用操作系统的标准输入输出进行通信                         │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                          │   │
│  │   Claude Desktop          subprocess             GitHub  │   │
│  │   (Host 进程)           (MCP Server)          Server   │   │
│  │                                                          │   │
│  │        │                        │                        │   │
│  │        │    stdin 管道        │                        │   │
│  │        ├──────────────────────►│                        │   │
│  │        │                        │                        │   │
│  │        │                        │   处理请求             │   │
│  │        │                        │                        │   │
│  │        │   stdout 管道         │                        │   │
│  │        │◄─────────────────────┤                        │   │
│  │        │                        │                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  消息格式：                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  {"jsonrpc": "2.0", "method": "tools/call", ...}    │   │
│  │  {"jsonrpc": "2.0", "id": 1, "result": {...}}        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  一行一条 JSON，通过换行符分隔                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么这样设计？**

```
┌─────────────────────────────────────────────────────────────────┐
│                    Stdio 的设计智慧                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 无需网络配置                                               │
│     ─────────────────────                                     │
│     本地进程间通信，不需要 IP、端口、证书                         │
│                                                                 │
│  2. 天然隔离                                                   │
│     ─────────────────────                                     │
│     subprocess 崩溃不影响主进程                                  │
│                                                                 │
│  3. 简单可靠                                                   │
│     ─────────────────────                                     │
│     操作系统提供的管道机制，稳定可靠                              │
│                                                                 │
│  4. 调试友好                                                   │
│     ─────────────────────                                     │
│     直接在终端看到输入输出                                       │
│                                                                 │
│  适用场景：                                                     │
│  • Claude Desktop 本地开发                                      │
│  • Cursor/Zed 本地集成                                         │
│  • 自动化测试                                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 HTTP + SSE 传输：远程通信方案

**场景**：当你需要让远程服务器运行 MCP 时。

```
┌─────────────────────────────────────────────────────────────────┐
│                   HTTP + SSE 传输原理                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  原理：HTTP 负责请求，SSE 负责推送                              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                          │   │
│  │   Client                              Server             │   │
│  │     │                                   │               │   │
│  │     │  1. HTTP POST /mcp              │               │   │
│  │     │  ─────────────────────────────►  │               │   │
│  │     │                                   │               │   │
│  │     │  2. HTTP 200 OK                 │               │   │
│  │     │  ◄────────────────────────────  │               │   │
│  │     │                                   │               │   │
│  │     │  3. 建立 SSE 连接 (可选)         │               │   │
│  │     │  ─────────────────────────────►  │               │   │
│  │     │                                   │               │   │
│  │     │  4. Server → Client 推送          │               │   │
│  │     │  ◄────────────────────────────  │               │   │
│  │     │     (text/event-stream)          │               │   │
│  │     │                                   │               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  关键概念：Server-Sent Events (SSE)                            │
│  ───────────────────────────────                                │
│  • 服务器 → 客户端的单向持久连接                                 │
│  • 自动重连机制                                                 │
│  • 简单文本协议                                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么需要两种协议？**

```
┌─────────────────────────────────────────────────────────────────┐
│                   HTTP + SSE 分工原理                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  HTTP (请求-响应)          │  SSE (服务器推送)                  │
│  ─────────────────────    │  ─────────────────────            │
│  Client 发起              │  Server 主动推送                   │
│  短连接                   │  持久连接                           │
│  适合单向请求             │  适合通知、进度                     │
│                           │                                     │
│  ┌─────────────────────┐ │  ┌─────────────────────┐        │
│  │ GET/POST 请求        │ │  │ data: {...}         │        │
│  │ → 获取工具列表      │ │  │ data: {...}         │        │
│  │ → 调用工具          │ │  │ event: progress     │        │
│  │ → 读取资源          │ │  └─────────────────────┘        │
│  └─────────────────────┘ │                                    │
│                           │                                     │
│  MCP 中用途：             │  MCP 中用途：                        │
│  • 工具调用              │  • 能力变更通知                     │
│  • 资源读取              │  • 日志推送                         │
│  • 提示获取              │  • 进度更新                         │
│                           │                                     │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 Streamable HTTP：现代高性能方案

**场景**：MCP 1.2 引入的新传输层，针对高并发优化。

```
┌─────────────────────────────────────────────────────────────────┐
│                Streamable HTTP 传输原理                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  核心思想：单一持久连接，双向流式通信                            │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                          │   │
│  │   Client                              Server            │   │
│  │     │                                   │               │   │
│  │     │  1. HTTP POST /mcp (初始化)     │               │   │
│  │     │  ─────────────────────────────►  │               │   │
│  │     │                                   │               │   │
│  │     │  2. 返回连接 ID + HTTP 200        │               │   │
│  │     │  ◄────────────────────────────  │               │   │
│  │     │                                   │               │   │
│  │     │  3. 同一个连接发送请求            │               │   │
│  │     │  ─────────────────────────────►  │               │   │
│  │     │                                   │               │   │
│  │     │  4. 流式响应 (分块传输)           │               │   │
│  │     │  ◄────────────────────────────  │               │   │
│  │     │     (Chunked Transfer)           │               │   │
│  │     │                                   │               │   │
│  │     │  5. 继续发送更多请求...           │               │   │
│  │     │  ─────────────────────────────►  │               │   │
│  │     │                                   │               │   │
│  │     │  6. 流式响应继续...              │               │   │
│  │     │  ◄────────────────────────────  │               │   │
│  │     │                                   │               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  对比传统 HTTP：                                                │
│                                                                 │
│  ┌────────────────┬─────────────┬─────────────────┐         │
│  │     维度        │  传统 HTTP  │  Streamable HTTP │         │
│  ├────────────────┼─────────────┼─────────────────┤         │
│  │  连接数        │  每个请求   │  单一持久连接   │         │
│  ├────────────────┼─────────────┼─────────────────┤         │
│  │  响应方式      │  一次性返回 │  流式传输      │         │
│  ├────────────────┼─────────────┼─────────────────┤         │
│  │  并发能力      │    中等     │     高         │         │
│  ├────────────────┼─────────────┼─────────────────┤         │
│  │  适用场景      │  简单 CRUD  │  高并发、AI    │         │
│  └────────────────┴─────────────┴─────────────────┘         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 安全考虑

```
┌─────────────────────────────────────────────────────────────────┐
│                    传输层安全建议                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Stdio:                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  • 最小权限运行                                          │   │
│  │  • 隔离文件系统访问                                      │   │
│  │  • 不以 root 运行                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  HTTP/SSE:                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  • 始终使用 HTTPS                                       │   │
│  │  • 实现 API Key 或 OAuth 认证                           │   │
│  │  • 添加速率限制                                          │   │
│  │  • CORS 配置正确                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Streamable HTTP:                                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  • 连接验证和会话管理                                    │   │
```
