## 一、引子：从本地函数到网络调用

### 1.1 同一个进程的调用

想象你正在编写一个天气应用。当用户输入城市名时，你的代码直接调用获取天气的函数：

```
┌─────────────────────────────────────────────────────────────────┐
│                       本地函数调用                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────┐                                         │
│   │   调用方代码    │                                         │
│   │                 │                                         │
│   │ weather =      │                                         │
│   │   get_weather( │                                         │
│   │     "北京"     │                                         │
│   │   )            │                                         │
│   └────────┬────────┘                                         │
│            │                                                   │
│            │ 直接调用（内存中）                                │
│            ▼                                                   │
│   ┌─────────────────┐                                         │
│   │   函数定义       │                                         │
│   │                 │                                         │
│   │ def get_weather │                                         │
│   │     (city):     │                                         │
│   │     return ...  │                                         │
│   └─────────────────┘                                         │
│                                                                 │
│   特点：                                                      │
│   • 零延迟                                                     │
│   • 类型安全（编译期检查）                                    │
│   • 调试简单（可直接断点）                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

这种调用方式简单高效，但在现代分布式系统中，我们经常需要：

**"让运行在 Claude Desktop 上的 AI，能够调用运行在服务器上的工具函数"**

这就是 RPC（远程过程调用）要解决的问题。

### 1.2 网络时代的挑战

当函数调用跨越网络时，情况变得复杂：

```
┌─────────────────────────────────────────────────────────────────┐
│                       远程调用困境                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Claude Desktop                 服务器                           │
│   ┌─────────────┐             ┌─────────────┐              │
│   │              │    网络     │              │              │
│   │ get_weather │◄──────────►│   Python    │              │
│   │   ("北京")  │   请求/响应 │   函数      │              │
│   │              │             │              │              │
│   └─────────────┘             └─────────────┘              │
│                                                                 │
│   挑战：                                                       │
│   ┌───────────────────────────────────────────────────────┐    │
│   │                                                       │    │
│   │  1. 如何序列化数据？（JSON？Protobuf？XML？）        │    │
│   │                                                       │    │
│   │  2. 如何表示"调用哪个函数"？                         │    │
│   │                                                       │    │
│   │  3. 如何处理错误？（网络断开？服务崩溃？）           │    │
│   │                                                       │    │
│   │  4. 如何保持状态？（每次请求独立？）                 │    │
│   │                                                       │    │
│   │  5. 如何支持异步？（不阻塞等待？）                   │    │
│   │                                                       │    │
│   └───────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**JSON-RPC 2.0 正是为了解决这些问题而设计的**。

## 二、JSON-RPC 2.0 核心规范

### 2.1 协议设计原则

在深入细节之前，理解协议的设计原则很重要：

```
┌─────────────────────────────────────────────────────────────────┐
│                    JSON-RPC 2.0 设计原则                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 简单性优先                                                 │
│     ─────────────                                             │
│     "少即是多"——用最少的特性解决核心问题                        │
│                                                                 │
│     只定义：                                                    │
│     • 如何表示"调用哪个方法"                                    │
│     • 如何传递参数                                              │
│     • 如何返回结果                                              │
│                                                                 │
│  2. 无状态设计                                                 │
│     ─────────────                                             │
│     每个请求都是独立的，服务不需要记住之前发生了什么              │
│                                                                 │
│     优点：                                                     │
│     • 易于水平扩展                                              │
│     • 易于负载均衡                                             │
│     • 易于故障恢复                                              │
│                                                                 │
│  3. 双向通信                                                   │
│     ─────────────                                             │
│     支持两种消息：                                              │
│     • Request（请求）- 需要响应                                 │
│     • Notification（通知）- 不需要响应                          │
│                                                                 │
│  4. 批处理支持                                                 │
│     ─────────────                                             │
│     一次发送多个请求，减少网络往返                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 请求与响应格式

这是 JSON-RPC 最基础的部分：

```
┌─────────────────────────────────────────────────────────────────┐
│                      请求格式                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  JSON 格式：                                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ {                                                         │   │
│  │   "jsonrpc": "2.0",     ← 协议版本（必须）          │   │
│  │   "method": "add",      ← 要调用的方法（必须）      │   │
│  │   "params": [5, 3],     ← 方法参数（可选）          │   │
│  │   "id": 1                ← 请求标识（必须）          │   │
│  │ }                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  字段说明：                                                     │
│  ┌──────────┬────────────┬─────────────────────────────────┐  │
│  │ 字段     │ 类型        │ 说明                           │  │
│  ├──────────┼────────────┼─────────────────────────────────┤  │
│  │ jsonrpc  │ string     │ 必须是 "2.0"                  │  │
│  ├──────────┼────────────┼─────────────────────────────────┤  │
│  │ method   │ string     │ 要调用的方法名                 │  │
│  ├──────────┼────────────┼─────────────────────────────────┤  │
│  │ params   │ array/obj  │ 方法参数，可省略               │  │
│  ├──────────┼────────────┼─────────────────────────────────┤  │
│  │ id       │ string/    │ 请求标识，                     │  │
│  │          │ number/    │ 用于匹配响应，                 │  │
│  │          │ null       │ Notification 时可为 null       │  │
│  └──────────┴────────────┴─────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```
┌─────────────────────────────────────────────────────────────────┐
│                      响应格式                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  成功响应：                                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ {                                                         │   │
│  │   "jsonrpc": "2.0",                                     │   │
│  │   "result": 8,            ← 调用的结果                 │   │
│  │   "id": 1                  ← 对应请求的 id             │   │
│  │ }                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  失败响应：                                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ {                                                         │   │
│  │   "jsonrpc": "2.0",                                     │   │
│  │   "error": {                                             │   │
│  │     "code": -32602,       ← 错误码                      │   │
│  │     "message": "Invalid params", ← 错误描述             │   │
│  │     "data": "..."         ← 额外信息（可选）           │   │
│  │   },                                                       │   │
│  │   "id": 1                  ← 对应请求的 id             │   │
│  │ }                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ⚠️ 注意：result 和 error 互斥，只能出现一个！               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 预定义错误码

JSON-RPC 定义了一组标准错误码：

```
┌─────────────────────────────────────────────────────────────────┐
│                     JSON-RPC 预定义错误码                         │
├──────────────┬────────────────────────────────────────────────┤
│    错误码     │                    说明                         │
├──────────────┼────────────────────────────────────────────────┤
│  -32700      │ Parse Error - JSON 解析失败                    │
├──────────────┼────────────────────────────────────────────────┤
│  -32600      │ Invalid Request - 无效的请求对象              │
├──────────────┼────────────────────────────────────────────────┤
│  -32601      │ Method Not Found - 方法不存在                  │
├──────────────┼────────────────────────────────────────────────┤
│  -32602      │ Invalid Params - 参数无效                      │
├──────────────┼────────────────────────────────────────────────┤
│  -32603      │ Internal Error - 内部错误                      │
└──────────────┴────────────────────────────────────────────────┘

额外说明：
┌─────────────────────────────────────────────────────────────────┐
│  -32000 到 -32099 保留给实现自定义错误                         │
│                                                                 │
│  常用扩展：                                                     │
│  • -32000: Server Error                                        │
│  • -32001: Resource Not Found                                  │
│  • -32002: Tool Not Found                                      │
└─────────────────────────────────────────────────────────────────┘
```

## 三、核心概念图解

### 3.1 请求-响应配对原理

理解 JSON-RPC 最重要的是理解"请求-响应配对"机制：

```
┌─────────────────────────────────────────────────────────────────┐
│                   请求-响应配对原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  id 是连接请求和响应的桥梁：                                    │
│                                                                 │
│  请求方                          响应方                         │
│  ┌─────────────┐              ┌─────────────┐              │
│  │             │              │             │              │
│  │ id: 1       │──────┐  ┌────│ id: 1       │              │
│  │             │      │  │    │             │              │
│  └─────────────┘      │  │    └─────────────┘              │
│                       │  │                                    │
│                       │  │  响应必须携带                     │
│                       │  │  相同的 id                        │
│                       │  │                                    │
│  ┌─────────────┐      │  │    ┌─────────────┐              │
│  │             │      │  │    │             │              │
│  │ id: 2       │──────┼──┼────│ id: 2       │              │
│  │             │      │  │    │             │              │
│  └─────────────┘      │  │    └─────────────┘              │
│                       │  │                                    │
│  ┌─────────────┐      │  │    ┌─────────────┐              │
│  │             │      │  │    │             │              │
│  │ id: null    │──────┴──┴────│ id: null    │              │
│  │ (notification)            (无响应)  │              │
│  └─────────────┘              └─────────────┘              │
│                                                                 │
│  三种 id 情况：                                                 │
│  ┌───────────────────────────────────────────────────────┐    │
│  │                                                       │    │
│  │  1. 有具体 id 值                                      │    │
│  │     • 期待响应                                        │    │
│  │     • 用于匹配响应                                      │    │
│  │                                                       │    │
│  │  2. id: null                                         │    │
│  │     • 这叫"通知"(Notification)                        │    │
│  │     • 不期待响应                                       │    │
│  │     • 用于单向事件通知                                 │    │
│  │                                                       │    │
│  │  3. 没有 id 字段                                      │    │
│  │     • 无效请求！                                       │    │
│  │                                                       │    │
│  └───────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 批量请求处理

JSON-RPC 支持批量发送请求，特别适合需要获取多个数据的情况：

```
┌─────────────────────────────────────────────────────────────────┐
│                      批量请求处理                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  场景：AI 需要同时获取多个信息                                   │
│                                                                 │
│  发送批量请求：                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ [                                                         │   │
│  │   {"jsonrpc": "2.0", "method": "get_user", "id": 1},  │   │
│  │   {"jsonrpc": "2.0", "method": "get_repos", "id": 2}, │   │
│  │   {"jsonrpc": "2.0", "method": "get_issues", "id": 3}  │   │
│  │ ]                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                       │                                         │
│                       ▼                                         │
│  服务器返回批量响应：                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ [                                                         │   │
│  │   {"jsonrpc": "2.0", "result": {...}, "id": 1},        │   │
│  │   {"jsonrpc": "2.0", "result": {...}, "id": 2},        │   │
│  │   {"jsonrpc": "2.0", "error": {...}, "id": 3}          │   │
│  │ ]                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                       │                                         │
│                       ▼                                         │
│  优势：                                                         │
│  • 一次网络往返获取多个数据                                     │
│  • 显著降低延迟                                                 │
│  • 服务端可以并行处理                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 四、MCP 方法映射

### 4.1 MCP 如何使用 JSON-RPC

MCP 协议实际上是 JSON-RPC 2.0 的一种"应用层协议"——它定义了在这个通道上传输哪些"方法"：

```
┌─────────────────────────────────────────────────────────────────┐
│                    MCP 核心方法定义                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  MCP 定义了一组标准方法：                                        │
│                                                                 │
│  ┌────────────────────┬─────────────────────────────────────┐ │
│  │      方法           │              说明                     │ │
│  ├────────────────────┼─────────────────────────────────────┤ │
│  │ initialize         │ 初始化连接，交换能力                 │ │
│  ├────────────────────┼─────────────────────────────────────┤ │
│  │ tools/list         │ 列出所有可用工具                     │ │
│  ├────────────────────┼─────────────────────────────────────┤ │
│  │ tools/call         │ 调用指定工具                         │ │
│  ├────────────────────┼─────────────────────────────────────┤ │
│  │ resources/list     │ 列出所有可用资源                     │ │
│  ├────────────────────┼─────────────────────────────────────┤ │
```

## 五、深入理解：为什么这样设计

### 5.1 为什么选择 JSON？

这是一个常见的问题：为什么 JSON-RPC 使用 JSON 而不是更高效的二进制格式？

```
┌─────────────────────────────────────────────────────────────────┐
│                    JSON vs 二进制对比                              │
├─────────────────────────────────┬────────────────────────────────┤
│          维度                   │   JSON        │   Protobuf    │
├─────────────────────────────────┼────────────────────────────────┤
│  可读性                       │   ✅ 高     │    ❌ 低       │
├─────────────────────────────────┼────────────────────────────────┤
│  调试便利                     │   ✅ 高     │    ❌ 低       │
├─────────────────────────────────┼────────────────────────────────┤
│  解析性能                     │   ❌ 中     │    ✅ 高       │
├─────────────────────────────────┼────────────────────────────────┤
│  序列化后大小                 │   ❌ 大     │    ✅ 小       │
├─────────────────────────────────┼────────────────────────────────┤
│  生态支持                     │   ✅ 极好   │    ✅ 好       │
├─────────────────────────────────┼────────────────────────────────┤
│  学习成本                     │   ✅ 低     │    ❌ 中       │
└─────────────────────────────────┴────────────────────────────────┘

答案：
JSON-RPC 主要用于"工具调用"场景，这类场景的特点是：
• 请求频率相对较低
• 数据量较小（参数和返回值）
• 可调试性非常重要
• 跨语言兼容性要求高

因此 JSON 是最优选择。
```

### 5.2 为什么是"无状态"的？

JSON-RPC 的无状态设计是有意为之的：

```
┌─────────────────────────────────────────────────────────────────┐
│                    有状态 vs 无状态                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  有状态协议（如 WebSocket 长期连接）：                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  Client ──────────► 建立连接 ──────────► Server        │   │
│  │    │                                                     │   │
│  │    │  保持连接，记住之前发生了什么                      │   │
│  │    │                                                     │   │
│  │    ▼                                                     │   │
│  │  请求1 ──► 处理中 ──► 响应1                            │   │
│  │    │                                                     │   │
│  │    ▼                                                     │   │
│  │  请求2 ──► [使用请求1的上下文] ──► 响应2               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  问题：                                                         │
│  • 难以扩展（需要粘性会话）                                      │
│  • 难以容错（连接断开丢失状态）                                  │
│  • 难以调试（难以复现问题）                                     │
│                                                                 │
│  ──────────────────────────────────────────────────────────    │
│                                                                 │
│  无状态协议（JSON-RPC）：                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  请求1 ──────────────────────────────────────────►      │   │
│  │  {id: 1, params: {...}, context: {...}}                │   │
│  │                                                         │   │
│  │  请求2 ──────────────────────────────────────────►      │   │
│  │  {id: 2, params: {...}, context: {...}}  ← 包含所有   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  优势：                                                         │
│  • 任意服务器处理任意请求                                        │
│  • 任意时刻容错恢复                                             │
│  • 完全可复现调试                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 六、最佳实践

### 6.1 错误处理原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    JSON-RPC 错误处理最佳实践                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 使用标准错误码                                             │
│     ┌─────────────────────────────────────────────────────┐   │
│  │  if not user:                                        │   │
│  │      return {                                        │   │
│  │          "jsonrpc": "2.0",                          │   │
│  │          "error": {                                  │   │
│  │              "code": -32602,     ← 格式错误          │   │
│  │              "message": "Invalid params: missing user" │   │
│  │          },                                          │   │
│  │          "id": request_id                           │   │
│  │      }                                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                                 │
│  2. 提供有意义的错误消息                                         │
│     ┌─────────────────────────────────────────────────────┐   │
│  │  ❌ 错误: {"message": "Error"}                        │   │
│  │  ✅ 错误: {"message": "User not found",             │   │
│  │           "data": {"user_id": "usr_123"}}           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                                 │
│  3. 区分客户端错误和服务端错误                                    │
│     • -32600 ~ -32699: 客户端错误（请求有问题）                │
│     • -32000 ~ -32099: 服务端错误（服务有问题）               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 批量请求使用场景

```
┌─────────────────────────────────────────────────────────────────┐
│                    批量请求使用场景                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  适合使用批量请求：                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  场景1: AI 需要多个工具的信息                            │   │
│  │  • 获取用户信息                                          │   │
│  │  • 获取用户仓库列表                                      │   │
│  │  • 获取最近活动                                          │   │
│  │  → 一次请求全部获取                                      │   │
│  │                                                         │   │
│  │  场景2: 初始化时的元信息获取                            │   │
│  │  • 列出工具                                             │   │
│  │  • 列出资源                                             │   │
│  │  • 列出提示模板                                         │   │
│  │  → 一次请求全部获取                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  不适合使用批量请求：                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  场景: 请求之间有依赖                                   │   │
│  │  • 请求1: 获取用户 ID                                   │   │

│  │  → 必须顺序执行，无法批量                               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
